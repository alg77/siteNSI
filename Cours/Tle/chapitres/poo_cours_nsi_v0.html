CHAPITRE 01 La classe : modéliser le réel
🔭 1.1 — Pourquoi la Programmation Orientée Objet ? Fondement

En programmation procédurale classique, on manipule des données séparées des fonctions qui les traitent. La POO regroupe données (attributs) et comportements (méthodes) dans une seule entité : l'objet.

Exemple concret : modéliser un compte bancaire.

✗ Sans POO — données éparpillées
titulaire = "Alice"
solde = 1500

def deposer(solde, montant):
    return solde + montant

def afficher(titulaire, solde):
    print(titulaire, solde)
# Données et fonctions séparées,
# rien n'empêche de mélanger
# des soldes de comptes différents
✓ Avec POO — tout regroupé
class Compte:
    def __init__(self, titulaire, solde):
        self.titulaire = titulaire
        self.solde = solde

    def deposer(self, montant):
        self.solde += montant

    def afficher(self):
        print(self.titulaire, self.solde)
# Données et comportements liés

CHAPITRE 02 Le constructeur __init__
🔧 2.1 — Rôle et syntaxe du constructeur Incontournable

Le constructeur est la méthode automatiquement appelée par Python au moment de la création d'un objet. Son rôle : initialiser les attributs de l'objet.

1
Nom imposé : __init__ Deux underscores avant et deux après. Ni plus ni moins. C'est une convention Python pour les méthodes spéciales.
2
Premier paramètre toujours self self représente l'objet en cours de création. Python le passe automatiquement — on n'écrit jamais self lors de l'appel.
3
Déclarer chaque attribut avec self.nom = valeur Sans self., la variable est locale et disparaît à la fin du constructeur.
4
Jamais de return Le constructeur ne retourne jamais rien. Python gère ça en interne. 
class Eleve:
    def __init__(self, nom, note):
        self.nom  = nom    # attribut "nom" accessible partout dans la classe
        self.note = note   # attribut "note"
        # Pas de return !

# Instanciation — on NE passe PAS self
e1 = Eleve("Alice", 15)
e2 = Eleve("Bob",   11)

print(e1.nom)    # "Alice"
print(e2.note)   # 11

👁️ 3.2 — Les accesseurs : lire sans modifier Getter 
class Rectangle:
    def __init__(self, l, h):
        self.largeur = l
        self.hauteur = h

    def aire(self):                        ← accesseur
        return self.largeur * self.hauteur  ← return obligatoire

    def perimetre(self):
        return 2 * (self.largeur + self.hauteur)

r = Rectangle(4, 6)
print(r.aire())       # 24  — appel avec ()
print(r.perimetre())  # 20  — appel avec ()
⚡ Piège — Attribut vs Méthode : les parenthèses
✗ Sans parenthèses — référence à la méthode
r.aire       # <bound method...>
             # C'est l'objet "méthode", pas le résultat

if r.aire > 10:  # TypeError !
    ...
✓ Avec parenthèses — appel, résultat retourné
r.aire()      # 24

if r.aire() > 10:  # True
    ...

Règle mnémotechnique : un attribut est une donnée → objet.attribut (pas de parenthèses). Une méthode est une action → objet.methode() (parenthèses obligatoires).

class Compte:
    def __init__(self, titulaire, solde):
        self.titulaire = titulaire
        self.solde = solde

    def deposer(self, montant):             ← mutateur, pas de return
        if montant > 0:
            self.solde = self.solde + montant

    def retirer(self, montant):
        if montant > 0 and self.solde >= montant:
            self.solde -= montant

    def get_solde(self):                    ← accesseur
        return self.solde

c = Compte("Alice", 1000)

🔄 4.2 — Parcourir : extraire et cumuler Pattern fondamental

Le pattern le plus courant au bac : accumulateur + boucle.

# ── Calculer la somme d'un attribut ──────────────────────────
def somme_notes(promo):
    total = 0                        ← accumulateur initialisé AVANT la boucle
    for eleve in promo:              ← eleve est un objet Eleve
        total = total + eleve.note   ← attribut → PAS de ()
    return total                     ← return APRÈS la boucle


# ── Calculer une moyenne ─────────────────────────────────────
def moyenne(promo):
    if len(promo) == 0:
        return 0                     # éviter la division par zéro
    total = 0
    for e in promo:
        total += e.note
    return total / len(promo)

# ── Construire une sous-liste filtrée ────────────────────────
def eleves_admis(promo):

CHAPITRE 05 Notions avancées (niveau Bac)
🔀 5.1 — Objets passés en paramètre d'une méthode Interaction d'objets

Une méthode peut prendre un autre objet de la même classe (ou d'une autre) en paramètre.

class Rectangle:
    def __init__(self, l, h):
        self.largeur = l
        self.hauteur = h

    def aire(self):
        return self.largeur * self.hauteur

    def est_plus_grand(self, autre):   ← autre est aussi un Rectangle
        return self.aire() > autre.aire()
        # aire() sur self ET sur autre — parenthèses des deux côtés !

r1 = Rectangle(4, 6)   # aire = 24
r2 = Rectangle(3, 9)   # aire = 27
print(r1.est_plus_grand(r2))  # False
🔒 5.2 — Encapsulation et getter/setter Bonne pratique

L'encapsulation consiste à protéger les attributs en les rendant accessibles uniquement via des méthodes dédiées (getters et setters), plutôt qu'en les modifiant directement de l'extérieur.

class Batterie:
    def __init__(self, charge):
        self.charge = charge   # 0 à 100

    def get_charge(self):         ← accesseur
        return self.charge

    def set_charge(self, val):    ← mutateur avec validation
        if 0 <= val <= 100:
            self.charge = val
        # si val invalide : on ignore (ou on lève une erreur)

b = Batterie(50)
b.set_charge(80)         # OK
b.set_charge(150)        # ignoré : hors plage
print(b.get_charge())    # 80
⚑ À retenir : 0 <= val <= 100 est une syntaxe Python valide pour enchaîner des comparaisons (équivalent à val >= 0 and val <= 100). Elle est plus lisible et conseillée.

CHAPITRE 06 Héritage et polymorphisme
🧬 6.1 — Pourquoi l'héritage ? Motivation

L'héritage permet de créer une nouvelle classe à partir d'une classe existante. La nouvelle classe (sous-classe ou classe enfant) hérite automatiquement de tous les attributs et méthodes de la classe d'origine (classe parent ou classe de base), et peut en ajouter ou en modifier.

C'est le principe du "est-un" : un PokemonFeu est un Pokemon — il a donc tout ce qu'un Pokemon a, plus des choses en plus.

Sans héritage
Réécrire nom, pdv, attaquer… dans chaque type de Pokémon → duplication massive
Avec héritage
Écrire une fois dans Pokemon, spécialiser seulement ce qui change dans chaque sous-classe
Classe parent
Pokemon
Attributs et méthodes communs à tous
Sous-classes
PokemonFeu, PokemonEau…
Ajoutent ou modifient le comportement
📐 6.2 — Syntaxe de l'héritage Syntaxe

Pour hériter d'une classe, on écrit le nom de la classe parent entre parenthèses après le nom de la sous-classe :

class Pokemon:                              ← classe parent
    def __init__(self, nom, type_p, pdv):
        self.nom  = nom
        self.type = type_p
        self.pdv  = pdv

    def attaquer(self, cible):
        cible.pdv -= 10
        print(f"{self.nom} attaque ! {cible.nom} a {cible.pdv} PV")

    def est_vaincu(self):
        return self.pdv <= 0


class PokemonFeu(Pokemon):                  ← hérite de Pokemon
    def __init__(self, nom, pdv):
        super().__init__(nom, "Feu", pdv)    ← appelle le constructeur parent
        self.flammes = 3                      ← attribut spécifique à PokemonFeu

    def attaque_flammes(self, cible):        ← méthode spécifique
        cible.pdv -= 20
        self.flammes -= 1
        print(f"{self.nom} lance Flammèche ! {cible.nom} perd 20 PV")


# Utilisation
salameche = PokemonFeu("Salamèche", 39)
pikachu   = Pokemon("Pikachu", "Électrique", 35)

salameche.attaquer(pikachu)        # méthode héritée de Pokemon
salameche.attaque_flammes(pikachu) # méthode propre à PokemonFeu
print(salameche.type)              # "Feu" — attribut hérité
⚑ À retenir : la sous-classe hérite automatiquement de toutes les méthodes du parent. On n'a pas besoin de les réécrire — salameche.attaquer() fonctionne sans qu'on l'ait définie dans PokemonFeu.
🔑 6.3 — super() : appeler le parent Mot-clé essentiel

super() est une fonction Python qui donne accès aux méthodes de la classe parent depuis la sous-classe. Son usage le plus courant est dans __init__ pour initialiser les attributs communs sans les réécrire.

✗ Sans super() — duplication
class PokemonEau(Pokemon):
    def __init__(self, nom, pdv):
        # On réécrit ce que Pokemon fait déjà
        self.nom  = nom
        self.type = "Eau"
        self.pdv  = pdv
        self.reservoir = 5
✓ Avec super() — délégation propre
class PokemonEau(Pokemon):
    def __init__(self, nom, pdv):
        # On délègue au parent
        super().__init__(nom, "Eau", pdv)
        self.reservoir = 5  # attribut spécifique

super() peut aussi appeler n'importe quelle méthode du parent, pas seulement __init__ :

class PokemonElectrique(Pokemon):
    def __init__(self, nom, pdv):
        super().__init__(nom, "Électrique", pdv)
        self.paralysie = False

    def attaquer(self, cible):           ← surcharge de la méthode parent
        super().attaquer(cible)           ← d'abord l'attaque normale (-10 PV)
        cible.paralysie = True           ← puis l'effet spécial
        print(f"{cible.nom} est paralysé !")
⚡ Piège — Oublier super().__init__()
✗ Faux — attributs parents jamais créés
class PokemonFeu(Pokemon):
    def __init__(self, nom, pdv):
        pass  # ou juste self.flammes = 3
        # self.nom, self.pdv n'existent PAS
# → AttributeError à la première utilisation
✓ Correct
class PokemonFeu(Pokemon):
    def __init__(self, nom, pdv):
        super().__init__(nom, "Feu", pdv)
        self.flammes = 3
🎭 6.4 — Le polymorphisme : même nom, comportements différents Polymorphisme

Le polymorphisme signifie qu'une même méthode peut se comporter différemment selon la sous-classe qui l'implémente. On dit qu'on surcharge (ou redéfinit) la méthode.

L'intérêt : on peut appeler pokemon.attaquer(cible) sur n'importe quel type de Pokémon — Python sait automatiquement quelle version utiliser.

class Pokemon:
    def attaquer(self, cible):
        cible.pdv -= 10                   ← comportement par défaut

class PokemonFeu(Pokemon):
    def attaquer(self, cible):            ← surcharge : remplace le parent
        cible.pdv -= 15                   # plus puissant
        print("Attaque de Feu !")

class PokemonEau(Pokemon):
    def attaquer(self, cible):
        cible.pdv -= 12
        print("Jet d'eau !")

# Polymorphisme en action — même appel, comportements différents
equipe = [PokemonFeu("Salamèche", 39), PokemonEau("Carapuce", 44)]
cible  = Pokemon("Pikachu", "Électrique", 35)

for p in equipe:
    p.attaquer(cible)   # appelle la bonne version selon le type réel de p
⚑ Règle : quand Python voit p.attaquer(cible), il regarde d'abord dans la classe réelle de l'objet p. S'il y trouve la méthode, il l'utilise. Sinon, il remonte à la classe parent. C'est la résolution de méthode (MRO).
⚡ Piège — Surcharger sans appeler super() quand on veut garder le comportement parent
✗ Faux — perd le comportement parent
class PokemonElectrique(Pokemon):
    def attaquer(self, cible):
        # On oublie l'attaque de base !
        cible.paralysie = True
        # Résultat : cible ne perd AUCUN PV
✓ Correct — étend le comportement
class PokemonElectrique(Pokemon):
    def attaquer(self, cible):
        super().attaquer(cible)  # -10 PV d'abord
        cible.paralysie = True  # puis paralysie
🔍 6.5 — Tester le type d'un objet : isinstance() Outil pratique

isinstance(objet, Classe) retourne True si l'objet est une instance de la classe ou d'une de ses sous-classes. C'est la façon propre de tester le type en POO.

salameche = PokemonFeu("Salamèche", 39)

isinstance(salameche, PokemonFeu)  # True  — c'est bien un PokemonFeu
isinstance(salameche, Pokemon)     # True  — un PokemonFeu EST AUSSI un Pokemon
isinstance(salameche, PokemonEau)  # False — ce n'est pas un PokemonEau

# Utilisation typique dans une boucle
for p in equipe:
    if isinstance(p, PokemonElectrique):
        print(f"{p.nom} peut paralyser !")
⚔️ 6.6 — Exemple complet : hiérarchie Pokémon (niveau TP) Synthèse

Voici la structure complète telle qu'attendue dans un TP de terminale, avec héritage, super(), surcharge et liste d'objets.

import random

# ── Classe de base ───────────────────────────────────────────
class Pokemon:
    def __init__(self, nom, type_p, pdv):
        self.nom  = nom
        self.type = type_p
        self.pdv  = pdv

    def attaquer(self, cible):
        cible.pdv = max(0, cible.pdv - 10)  # pdv ne descend pas sous 0

    def est_vaincu(self):
        return self.pdv <= 0

    def soigner(self, soin):
        self.pdv += soin

    def __str__(self):
        return f"{self.nom} [{self.type}] — {self.pdv} PV"


# ── Sous-classes ─────────────────────────────────────────────
class PokemonElectrique(Pokemon):
    def __init__(self, nom, pdv):
        super().__init__(nom, "Électrique", pdv)

    def attaquer(self, cible):
        super().attaquer(cible)               # attaque normale d'abord
        if random.random() < 0.3:           # 30% de chances d'attaque spéciale
            super().attaquer(cible)           # double dégâts
            print(f"⚡ Tonnerre de {self.nom} !")

class PokemonFeu(Pokemon):
    def __init__(self, nom, pdv):
        super().__init__(nom, "Feu", pdv)

    def attaquer(self, cible):
        cible.pdv = max(0, cible.pdv - 15)  # plus puissant que la base
        if random.random() < 0.25:
            print(f"🔥 Flammèche de {self.nom} !")
            cible.pdv = max(0, cible.pdv - 10)

class PokemonEau(Pokemon):
    def __init__(self, nom, pdv):
        super().__init__(nom, "Eau", pdv)

    def attaquer(self, cible):
        cible.pdv = max(0, cible.pdv - 12)


# ── Fonction de duel ─────────────────────────────────────────
def duel(p1, p2):
    print(f"⚔️  {p1.nom} vs {p2.nom}")
    while not p1.est_vaincu() and not p2.est_vaincu():
        p1.attaquer(p2)
        if not p2.est_vaincu():
            p2.attaquer(p1)
    gagnant = p1 if not p1.est_vaincu() else p2
    print(f"🏆 {gagnant.nom} gagne !")
    return gagnant
